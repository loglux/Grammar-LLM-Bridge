## Пропускная способность

### При нормальной работе DeepSeek (latency 3-5s)

| Тип текста | Токены | Латентность | Throughput | Ограничение |
|------------|--------|-------------|------------|-------------|
| Короткий   | 100-200 | 1-2s | 40-60 req/min | DeepSeek latency |
| Средний    | 500-1000 | 3-5s | 20-30 req/min | **Норма работы** |
| Длинный    | 2000+ | 5-10s | 10-15 req/min | Таймауты возможны |

**Одновременно:** 8 запросов (по числу workers)
**Очередь:** Неограниченная (в Nginx → Uvicorn)

### При перегрузке DeepSeek (latency 30-60s)

⚠️ **КРИТИЧНО:** При высокой нагрузке на серверы DeepSeek:
- Запросы висят в очереди
- Workers блокируются на `await`
- Пропускная способность падает до **5-10 req/min**
- UX деградирует (ожидание 30-120s)

## Главное ограничение: DeepSeek API

### Из официальной документации DeepSeek:

> **DeepSeek API does NOT constrain user's rate limit.**
> However, when servers are under high traffic pressure, requests may take time to receive a response. During this period, the connection remains open and you may continuously receive empty lines (non-streaming) or SSE keep-alive comments (streaming).
> If the request has not started inference after 10 minutes, the server will close the connection.

### Что это означает:

✅ **Нет жёстких rate limits** (429 errors)
⚠️ **НО:** Качество сервиса зависит от нагрузки на DeepSeek
❌ **Риск:** При перегрузке латентность растёт, throughput падает

### Поведение при очереди на DeepSeek:

1. Клиент → Nginx → Backend → DeepSeek
2. DeepSeek в очереди → шлёт пустые строки (keep-alive)
3. httpx держит соединение (до 120s timeout)
4. При timeout → retry (до 2 раз)
5. **Итого:** до 6 минут ожидания на запрос!

## Узкие места (по приоритету)

### 1. 🔴 Качество сервиса DeepSeek (КРИТИЧНО)

**Проблема:**
- Непредсказуемая латентность (зависит от их нагрузки)
- При перегрузке: workers блокируются → throughput падает
- Нет контроля с нашей стороны

**Решение:**
- ✅ Внедрить fallback модель (GPT-4o-mini)
- ✅ Мониторить латентность (alert при >10s)
- ✅ Уменьшить LLM_TIMEOUT до 60s (быстрее failover)

### 2. ⚠️ Количество workers (8 total)

**Текущее состояние:**
- 2 инстанса × 4 workers = 8 одновременных обработчиков
- RAM/CPU использование <1% (можно увеличить)

**НО:**
- Увеличение workers НЕ поможет при проблемах с DeepSeek
- Больше workers = больше запросов → хуже качество DeepSeek
- **Рекомендация:** Оставить как есть до внедрения fallback

### 3. 💡 Один провайдер (только DeepSeek)

**Риск:**
- При проблемах с DeepSeek → весь сервис деградирует
- Нет резервного канала

**Решение:**
- Fallback на GPT-4o-mini (см. MULTI_LLM_DESIGN.md)

- Fallback можно отложить

- При перегрузке DeepSeek сервис станет непредсказуемым

- Настроить мониторинг латентности

**Шаги:**
1. Реализовать fallback (MULTI_LLM_DESIGN.md, фаза 1)
2. Настроить alert при latency >10s
3. Уменьшить LLM_TIMEOUT до 60s
4. Опционально: увеличить workers до 8 на инстанс

### Сценарий 3: Публичный сервис (10+ пользователей)

🔴 **КРИТИЧНО - Полная реализация multi-LLM**
- Fallback + language-specific routing
- Добавить 3-й инстанс (12 workers total)
- Мониторинг: Prometheus + Grafana
- Auto-scaling на базе очереди

**Шаги:**
1. Fallback модель (фаза 1)
2. Language-specific routing (фаза 2)
3. Увеличить инстансы до 3
4. Workers: 6-8 на инстанс (18-24 total)

## Известные ограничения

1. **DeepSeek API latency непредсказуема**
   - Зависит от их трафика
   - Решение: Fallback модель

2. **Нет auto-scaling**
   - Фиксированное число инстансов
   - Решение: Мануальное масштабирование при росте нагрузки

3. **httpx vs OpenAI SDK**
   - httpx может некорректно обработать много пустых строк от DeepSeek
   - Не наблюдалось на практике, но риск есть
   - Решение: Мониторить JSON parsing errors

4. **Single point of failure**
   - Nginx балансировщик в одном экземпляре
   - При падении nginx → весь сервис недоступен
   - Решение: Для критичных систем - keepalived + 2 nginx

## Часто задаваемые вопросы

**Q: Можно ли увеличить workers до 16-32?**
A: Технически да (RAM позволяет), но бессмысленно без fallback. При проблемах с DeepSeek все workers заблокируются, throughput упадёт независимо от их количества.

**Q: Сколько пользователей выдержит текущая конфигурация?**
A: 3-5 активных пользователей (Obsidian auto-check каждые 5s). При большем количестве внедряйте fallback.

**Q: Что делать при постоянных таймаутах?**
A: 1) Проверить статус DeepSeek API, 2) Временно переключить всех на GPT-4o-mini, 3) Внедрить fallback для автоматического переключения.

**Q: Нужно ли добавлять третий инстанс?**
A: Только если мониторинг показывает постоянную нагрузку >40 req/min и fallback уже внедрён. Иначе бесполезно.
